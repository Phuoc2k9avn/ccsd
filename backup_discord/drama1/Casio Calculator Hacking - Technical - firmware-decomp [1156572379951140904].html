The Administrator 9/27/2023 8:41 PM
In order to create an accurate decomp, the code will need to be separated into 4 categories: 1. Code written by CASIO in ANSI-C (main, diagnostic, reset_all, etc) 2. Code written by CASIO in ASM 3. Code from the SDK .lib files written by LAPIS (memset, etc) 4. Code from the SDK .asm files written by LAPIS (startup code, etc)
The Administrator 9/27/2023 8:51 PM
Code written by CASIO in ANSI-C will be decompiled by hand into ANSI-C code that can be compiled with CCU8 to produce matching assembly Code written by CASIO in ASM will be disassembled and reverse engineered Code from the U8Dev Tools (memset, etc) will most likely not need to be decompiled The entry/startup code (CE01 CEFF 0Ann) will most likely need to be recreated by modifying the very similar publicly available startup code .asm file of another u8 microcontroller as the startup code for the fx-ES PLUS microcontroller is not included in the U8Dev tools Likewise, other code specifically for the microcontroller may need to be disassembled or decompiled as it is also not publicly available. the .dcl file for the microcontroller will eventually also need to be recreated as it is not included in the U8Dev tools
8:56 PM

another way to identify CCU8 generated code: CCU8 has an /Orp option, which will enable use of register push/pop subroutines (ENTER/__regpushu8 and LEAVE/__regpopu8) on pre-classwiz, this option is turned off (which is default), however from classwiz onwards (incl. 2nd edition models) it is on. only CCU8 would actually use these push and pop registers (unless these functions are used in assembly code which is pretty unlikely), so it is same to assume that: if any function uses ENTER/__regpushu8 and LEAVE/__regpopu8 it is CCU8-generated code.
